# [머니로그] 프로젝트 관리 계획서

## 1. 서론

### 1.1 문서 목적 및 범위
* 본 프로젝트 계획서는 **머니로그(MoneyLog)** 개발에 필요한 전반적인 사항을 정의하여, 프로젝트를 효율적으로 진행하기 위한 지표로 삼는 데 목적이 있습니다.
* 이 문서는 프로젝트 개요, 개발 계획, 팀 구성, 품질 관리, 산출물 정의 등 개발 전 과정을 범위로 합니다.

### 1.2 프로젝트 개요
* 자산 관리의 가장 기본인 '기록'에 집중하되, 외부 은행 연동 없이 오직 사용자 기기 내부에만 데이터를 저장하여 개인정보를 완벽히 보호하는 심플한 모바일 앱을 제작합니다.

### 1.3 프로젝트 목표
* **사용자 주도 기록**: 사용자가 직접 수입/지출을 입력하게 하여 소비 흐름을 명확히 인지하도록 합니다.
* **프라이버시 강화**: 서버 통신을 배제한 로컬 저장 방식을 통해 데이터 유출 리스크를 원천 차단합니다.
* **소비 습관 개선**: 한눈에 들어오는 월간 통계 리포트를 통해 사용자의 합리적인 경제 활동을 지원합니다.

### 1.4 용어 정의
* **로컬 DB (Local Database)**: 데이터베이스를 외부 클라우드가 아닌 사용자 스마트폰 내부 저장소에 구축하는 방식입니다.
* **수동 입력 (Manual Entry)**: 은행/카드사 자동 연동 없이 사용자가 직접 금액과 날짜를 기입하는 방식입니다.

## 2. 개발 계획

### 2.1 개발 절차 모형
* 본 프로젝트는 요구사항이 명확해짐에 따라 유연하게 대처할 수 있도록 **Agile 방법론 중 SCRUM 모델**을 적용합니다. 
* 1인 개발 특성에 맞춰 주기적인 스프린트와 자체 리뷰를 통해 기능을 점진적으로 완성해 나갑니다.

### 2.2 WBS (Work Breakdown Structure)
프로젝트의 전체 목표를 달성하기 위해 다음과 같이 작업을 분할하여 관리합니다:
1. **요구분석**: 요구사항 수집 및 정의서 작성
2. **설계**: UI 화면 설계 및 로컬 DB 스키마 상세 설계
3. **구현**: 입력 기능, 조회 필터, 통계 리포트 엔진 코딩
4. **테스트**: 단위 및 전체 시스템 테스트, 매뉴얼 작성

### 2.3 개발 내용
| 개발 활동 | 설명 |
| :--- | :--- |
| **요구사항 분석** | 사용자가 필요로 하는 수입/지출 카테고리 및 통계 항목 정의 |
| **Product Backlog** | 개발할 기능들을 우선순위에 따라 나열 |
| **Sprint** | 1~2주 단위로 실제 동작하는 기능을 개발 |
| **Self-Review** | 스프린트 종료 시 개발된 기능의 완성도 스스로 평가 |

## 4. 품질 관리

### 4.1 개발 점검 계획 (기존 팀 미팅 대체)
1인 프로젝트이므로 협업 회의 대신 다음과 같은 자기 점검 활동을 수행합니다:
* **주간 계획 (Sprint Planning)**: 매주 월요일, 당일 주차에 완료할 목표 설정 
* **일일 점검 (Daily Check)**: 매일 작업 종료 전 진행률 확인 및 로그 기록
* **회고 (Retrospective)**: 스프린트 종료 후 잘된 점과 개선점 도출

### 4.2 변경 사항 관리 방법
기능 변경이 필요할 경우 반드시 문서에 기록하고 Git 커밋 이력을 통해 전/후 과정을 관리합니다. 변경 사항이 프로젝트 일정에 미치는 영향을 검토하여 우선순위를 재조정합니다.

### 4.3 산출물 관리 방법
* **명명 규칙**: '[머니로그]문서명_YYMMDD_Doc-버전' 형식을 적용합니다.
* **저장 관리**: 소스 코드는 **GitHub**에 저장하고, 문서는 클라우드 서버에 백업하여 유실을 방지합니다.
